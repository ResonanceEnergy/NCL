# NCL — Entropy-Aware Memory Layer (v1.0)

## Problem
Memory systems accumulate entropy as:
- Outdated notes
- Duplicated facts
- Broken links
- Unclear provenance
- Confidence inflation
- Context loss

## Solution: Entropy-Aware Memory
Every note has:
- Freshness state
- Confidence level
- Decay timer
- Refresh trigger
- Pruning rule

---

## States
| State | Description |
|-------|-------------|
| Fresh | Captured recently, high context |
| Stable | Verified or repeatedly reinforced |
| Degrading | Likely stale, needs refresh |
| Obsolete | Conflicts with newer evidence or reality |

## Rules
1. Notes decay unless refreshed by:
   - Evidence upgrade
   - Repeated use
   - Successful prediction / outcome alignment

2. Contradictions create a "fork":
   - Keep both claims
   - Label conflict
   - Schedule resolution experiment

3. Summaries must be multi-scale (fractal memory):
   - **L0:** Raw capture (image, quote, thought, observation)
   - **L1:** Pattern note (one pattern, one claim, one implication)
   - **L2:** Domain synthesis (nature / mind / AI / org / markets)
   - **L3:** Doctrine delta candidate (what changes in our principles?)

---

## Entropy Valves for NCL
| Valve | Cadence | Action |
|-------|---------|--------|
| Weekly Prune | Weekly | Delete/merge duplicates |
| Monthly Refresh | Monthly | Re-validate key claims |
| Contradiction Resolver | On detection | Fork + schedule resolution |
| Confidence Decay | Continuous | Auto-downgrade stale notes |

---

## Cross-System Wiring
- **NCL captures →** BRS processes
- **BRS outputs →** NCC governs
- **NCC approves →** AAC operationalizes (markets)
- **Outputs come back →** NCL as memory

This loop is itself fractal.
