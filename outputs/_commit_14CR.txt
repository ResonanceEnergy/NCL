F. Scheduler typo fix (pre-existing bug surfaced during 14CR test)
   runtime/autonomous/scheduler.py _trend_tracker_loop
     - Was calling self._interruptible_sleep(90) — method doesn't
       exist on this class. Crashing the trend-tracker loop on
       every boot (supervisor was restarting it 3x then giving up).
     - Replaced with asyncio.sleep(90) matching every other loop
       boot-delay pattern in this file.

G. brief_archiver memory + tracker → fire-and-forget
   runtime/intelligence/brief_archiver.py
     - _persist_to_memory and _register_trade_ideas now run as
       background tasks via loop.create_task
     - Empirical finding: brain.memory_store.create_unit can
       stall indefinitely when the brief itself is still holding
       brain-internal locks. Decoupling = HTTP response returns
       in 0.0s instead of hanging the call.
     - Memory bg has a 30s timeout so it can't hang a task forever
     - Fixed _persist_to_memory signature to use real create_unit
       contract (content=, source=, importance=, tags=,
       memory_type=, metadata=) — previous version imported from
       nonexistent runtime.memory.memory_unit and silently failed
       (Wave 14Z + 14CQ both had this bug)
     - Response envelope: memory_unit_id_pending + ideas_to_register
       fields signal "check brain log [brief_archiver/bg] for outcome"
   Live verification end-to-end:
     archive_brief returns in 0.0s
     bg task: "registered 5 trade_ideas" 100ms later
     bg memory: still in flight at log-check time but capped 30s
