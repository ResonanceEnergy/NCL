# 2. Audit #7 — risk_governor NAV=$0 → halt-band race

`runtime/portfolio/auto_trader/loop.py:217-227` correctly guards
against the portfolio sync race: when bucket reads band=halt
AND NAV < $100, the loop coerces band="unknown" and clears the
drawdown_halt flag. But `runtime/portfolio/risk_governor.py:228`
was independently re-reading the bucket in the same tick →
saw band=halt → returned REJECT before the loop's clear took
effect. 26/106 historical rejects were this race.

Fix in `runtime/portfolio/risk_governor.py`:
  - check_proposed_trade gains a `band_override=` kwarg
  - When band_override is None (legacy callers), apply the
    same NAV<$100 guard inline before the band-halt branch
    decides REJECT. Reset multiplier to 1.0 alongside the band
    coercion so no haircut sneaks through.
  - When band_override is set, use it verbatim (caller knows
    best).

Fix in `runtime/portfolio/auto_trader/loop.py:365-371`:
  - Pass `band_override=band` into check_proposed_trade so the
    loop's normalized "unknown" band wins.

After this, governor + loop see the SAME band per tick. Race
window closed.
