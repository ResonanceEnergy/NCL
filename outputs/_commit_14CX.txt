# 2. Night Watch HTTP 400 (audit fix #2)

runtime/autonomous/night_watch/analyst.py

  ROOT CAUSE: Python operator-precedence bug at line 535-544.
  The cost_analysis prompt used a ternary across the entire
  multi-line string:
    "...prompt...\n\n"
    f"DETERMINISTIC ISSUES FOUND:\n" + "\n".join(deterministic_issues[:10])
    if deterministic_issues
    else "None"

  When deterministic_issues was empty (the common case), the
  WHOLE prompt collapsed to the literal 4-char string "None".
  Anthropic accepted it but the response was useless.

  FIX: extracted _det_block ternary local to the issues line;
  prompt now always has the full operations-analyst context.

  Plus three defensive guards:
    - _call_anthropic enforces prompt_len >= 16 with a
      diagnostic fallback if shorter, so we never send naked
      "None" or whitespace.
    - resp.text logged on every 4xx (was raise_for_status with
      no body — future 400s show Anthropic's actual reason).
    - anthropic-version pinned with a Wave 14CX note for future
      bumps.

# 3. MOVES per-source divisor (audit fix #3)

runtime/intelligence/trend_tracker.py

  ROOT CAUSE: Wave 14CR's "days_present divisor" was computed
  ONCE across all buckets. So if Reddit has 4 days of history
  but youtube_search has only ~1 day (Wave 14CI reset its
  queries May 31), both got div_7d = min(7.0, 4.0) = 4.0. For
  youtube_search this means baseline ≈ cur_count/4 → ratio
  always ≈ 4.0. Audit Wave 14CW found all 23 youtube_search
  alerts identical at ratio=4.52.

  FIX: build src_earliest dict per source from the buckets
  iter, then _src_divisors(source) returns the correct
  divisor for each (source, ticker) pair.

  Live verified:
    BEFORE: 23 youtube_search alerts, 1 unique ratio (4.52)
    AFTER:  0 youtube_search alerts (data too short to fire
            any spiking/anomaly thresholds — correct behavior
            for a 1-day window).
    reddit: 16 alerts, 15 unique ratios (was identical-pair
            cluster pre-fix).
