Intel→Memory bridge: pin endpoint accepts JSON body + promote-if-missing

POST/DELETE /memory/working-context/pin now accept an optional JSON body
in addition to the legacy item_id query param. On POST, if the referenced
item_id isn't already in working context (typical for an Awarebot signal
that only lives in tier buffers), the endpoint falls back to
promote_item(content, source, tags, item_id) which inserts the item with
pinned=True. Pinned items survive day rollover via the existing
_carry_forward_pinned() path. DELETE is idempotent.

This unblocks the iOS Intel→promote-to-memory feature shipped today.
Signal item_ids use the existing convention: signal:<signal_id>.
