YTC stale fix: dedup window 7d->1d, BRAIN_AUTH_TOKEN falls back to STRIKE_AUTH_TOKEN

The YTC loop was firing every hour as designed but producing "no report" for
~10 hours because the 7-day video-ID dedup window was exhausted by the small
14-channel list. Every hourly run scraped ~40 candidates, all 40 matched the
47 already-analyzed IDs from the past 7 days, so 0 remained.

Fix 1 (runtime/councils/runner.py:215):
- _load_previously_analyzed_video_ids() default window 7d -> 1d
- Override via NCL_YTC_DEDUP_DAYS env var
- After this fix, videos analyzed >24h ago can be re-processed if scraped again,
  giving the loop fresh material every hour even with a small channel list

Fix 2 (runtime/councils/runner.py:53):
- BRAIN_AUTH_TOKEN now falls back to STRIKE_AUTH_TOKEN (the canonical name).
  Every YTC fire was logging "BRAIN_AUTH_TOKEN not set" — separate env var
  collision, fixed by aliasing.

Diagnosis surfaced in audit: YTC loop fires correctly on 1h cadence, but the
output stream was being silently suppressed by overly-aggressive dedup. Now
the iOS YTC + Brief->Council tabs will get fresh reports every hour.
